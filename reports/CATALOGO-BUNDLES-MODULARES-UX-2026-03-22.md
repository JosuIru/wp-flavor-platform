# Catalogo de Bundles Modulares UX

**Fecha:** 2026-03-22
**Objetivo:** Traducir el ecosistema de modulos del plugin a bundles coherentes de despliegue y experiencia de usuario.

---

## 1. Principio

El plugin no debe presentarse como una lista plana de 66 modulos.

Debe poder ofrecerse como combinaciones modulares coherentes segun:

- tipo de cliente
- tipo de comunidad
- nivel de complejidad
- necesidades de conversacion, gestion, economia, aprendizaje o red

Cada bundle define:

- un contexto principal
- uno o varios modulos base
- modulos nucleares
- modulos recomendados
- modulos transversales
- riesgos UX si se activa mal

## 4. Reglas de Activacion

### Regla 1

Primero se activa el contexto base.

### Regla 2

Despues se activan modulos nucleares.

### Regla 3

Solo despues se anaden modulos transversales.

### Regla 4

La capa federada o de red se activa al final.

### Regla 5

No se deberian activar muchos modulos del mismo ecosistema si no existe una narrativa UX clara para diferenciarlos.

---

## 5. Recomendaciones Operativas

### Bundle minimo viable

Para la mayoria de despliegues:

- 1 modulo base
- 2 o 3 nucleares
- 1 o 2 transversales

### Bundle medio

- 1 o 2 modulos base
- 4 o 6 nucleares
- 2 o 4 transversales

### Bundle complejo

- varios modulos base
- varios ecosistemas
- capa de red federada

En ese caso ya hace falta diseño de portal por contexto y no solo menu por modulo.

---

## 6. Conclusion

La mejor forma de explicar, vender y desplegar el plugin no es:

- "tenemos 66 modulos"

Sino:

- "tenemos bundles modulares por tipo de comunidad o cliente"

Y cada bundle puede enriquecerse con:

- conversacion
- contenido
- gestion
- infraestructura
- red

Este catalogo deberia servir para:

- priorizar activaciones
- definir perfiles de portal
- preparar demos
- crear onboarding por contexto
- evitar combinaciones incoherentes

---

## 7. Conexion con el Plan de Ejecucion

Para que estos bundles sean operables de verdad, la secuencia recomendada es:

### Fase P1

- cerrar jerarquia oficial
- cerrar rutas canonicas
- fijar reglas de relacion y contexto

### Fase P2

- crear helpers de rutas y contexto
- crear renderer comun de bloques relacionales

### Fase P3

- aplicar primero a los bundles mas rentables:
  - `Comunidad Base`
  - `Gobernanza Cooperativa`
  - `Economia Comunitaria`

### Fase P4

- integrar capas transversales y red federada
- ocultar complejidad no prioritaria

### Orden de bundles recomendado para ejecucion

1. `Comunidad Base`
2. `Gobernanza Cooperativa`
3. `Economia Comunitaria`
4. `Ayuntamiento / Portal Civico`
5. `Cuidados y Barrio`
6. `Aprendizaje y Cultura`
7. `Coworking y Espacios`
8. `Red Federada`

El criterio de ese orden es:

- primero hubs y modulos ya muy trabajados
- despues bundles que reutilizan mas infraestructura existente
- por ultimo despliegues complejos o federados
